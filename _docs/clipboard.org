#+AUTHOR: Kyle George (w/ Claude Code assistance)

* Overview

The dotfiles clipboard system is built around two portable scripts, =clip-in=
and =clip-out= (in =_bin/=), which abstract platform-specific clipboard access
behind a single interface.  Everything else -- tmux, vim, fzf, =clipcmd= --
shells out to these scripts, so platform support is centralized in one place.

tmux is configured with OSC 52 (=set-clipboard on=) and =allow-passthrough=,
which enables clipboard propagation through the terminal itself, independent
of =clip-in=/=clip-out=.  This is especially important for remote/SSH sessions.

* X11 Selections: CLIPBOARD vs PRIMARY

X11 has two independent selection buffers that are often confused:

** CLIPBOARD

The explicit clipboard, accessed via =Ctrl-C= / =Ctrl-V= in most GUI apps.
This is what =clip-in=/=clip-out= target, what vim's ="+= register uses, and
what OSC 52 sets.  It persists until something else overwrites it.

** PRIMARY

An implicit selection set automatically whenever you highlight text with the
mouse.  Pasted by middle-click (button 2).  It's ephemeral -- it changes
every time you select anything, even in a different window.

PRIMARY is useful for quick copy-paste without keyboard shortcuts, but it's
also easy to accidentally overwrite.  Most clipboard integration (=clip-in=,
tmux copy-pipe, OSC 52) deliberately targets CLIPBOARD only, because PRIMARY
is transient and better left to the mouse workflow.

** How the two interact in practice

- Select text with mouse → sets PRIMARY (paste with middle-click)
- =Ctrl-C= or =<leader>y= in vim → sets CLIPBOARD (paste with =Ctrl-V= or =<leader>p=)
- These are independent: setting one does not affect the other
- vim's ="*= register is PRIMARY; ="+= register is CLIPBOARD

** Wayland

Wayland has a single clipboard (equivalent to X11's CLIPBOARD), accessed via
=wl-copy=/=wl-paste=.  There is also a "primary selection" protocol but it's
optional and not all compositors implement it.  =wl-copy --primary= /
=wl-paste --primary= work on compositors that support it (GNOME does as of
GNOME 41+).

In practice, Wayland's clipboard behaves like X11's CLIPBOARD, and
mouse-select-to-copy works similarly to PRIMARY where the compositor supports
it.

** Windows / WSL

Windows has a single clipboard.  There is no primary selection concept.

* Mouse Selection and Clipboard

Mouse selection behavior depends on which layer handles the mouse event:

** Terminal handles mouse (tmux mouse OFF, or no tmux)

When tmux mouse mode is off (=prefix M=), or when not in tmux, the *terminal
emulator* handles mouse events directly:

| Action              | Terminal behavior                                      |
|---------------------+-------------------------------------------------------|
| Click-drag          | Selects text in the terminal; copies to PRIMARY (X11) |
|                     | or Windows clipboard (PuTTY, Windows Terminal)        |
| Shift + click-drag  | Same (forces terminal selection even if tmux mouse on)|
| Middle-click        | Pastes from PRIMARY (X11) or clipboard (PuTTY/WSL)   |
| Right-click         | Paste (PuTTY) or context menu (most Linux terminals)  |
| Ctrl-Shift-C        | Copy to CLIPBOARD (Linux terminals)                   |
| Ctrl-Shift-V        | Paste from CLIPBOARD (Linux terminals)                |

Terminal selection is raw terminal text -- it doesn't know about tmux panes,
so it can select across pane boundaries (usually not what you want).

*** PuTTY-specific mouse behavior

PuTTY always handles mouse selection itself (there's no passthrough to tmux
unless tmux mouse mode is on):

- Left-click drag: selects text, copies to Windows clipboard immediately
- Double-click: selects a word
- Triple-click: selects a line
- Right-click: pastes from Windows clipboard
- Middle-click: extends selection (not paste, unlike X11)

To select text while tmux mouse mode is on, hold *Shift* while clicking/
dragging -- this forces PuTTY to handle the selection instead of tmux.

*** Windows Terminal mouse behavior

- Left-click drag: selects text, does NOT auto-copy (must =Ctrl-Shift-C=)
- Right-click: pastes from clipboard (default; configurable)
- Shift + click-drag: forces terminal selection when tmux mouse is on

** tmux handles mouse (mouse ON -- default)

When tmux mouse mode is on (=set -g mouse on=, the default), tmux intercepts
mouse events.  Selection is *pane-aware* -- it only selects within the current
pane, which is usually what you want.

| Action              | tmux behavior                                        |
|---------------------+------------------------------------------------------|
| Click-drag          | Enters copy mode, selects text within pane           |
| Release mouse       | Copies selection to system clipboard via =clip-in=   |
|                     | (configured by =MouseDragEnd1Pane= binding)          |
| Middle-click        | Not handled by tmux (falls through to terminal)      |
| Shift + click-drag  | Bypasses tmux -- terminal handles selection directly |

tmux mouse selection uses the =MouseDragEnd1Pane= binding (=tmux.conf=),
which switches between =copy-pipe-no-clear= and =copy-pipe-and-cancel=
based on whether the user was already scrolled back before the drag:

#+begin_example
bind-key -T copy-mode-vi MouseDragEnd1Pane {
    if-shell -F '#{scroll_position}' {
        send-keys -X copy-pipe-no-clear 'clip-in'
    } {
        send-keys -X copy-pipe-and-cancel 'clip-in'
    }
}
#+end_example

- If =scroll_position == 0= (user was at the prompt; the drag is what
  entered copy mode): exit copy mode after copying so the cursor stays at
  the prompt.
- If =scroll_position > 0= (user was already in copy mode from scrolling
  back): stay in copy mode so they can keep navigating; press =q= to
  exit.

This avoids the "cursor jumps back to the shell prompt" UX when scrolled
back, *and* avoids the "stays in copy mode unnecessarily" UX when just
selecting at the prompt.  The keyboard yank bindings (=y=, =M-y=) still
use =copy-pipe-and-cancel= unconditionally because pressing a key to copy
*implies* you're done navigating.

Mouse-selected text goes through =clip-in= → system clipboard (and also
the tmux buffer).  With OSC 52, this reaches the system clipboard even
over SSH -- provided the outer terminal supports OSC 52 (Terminal.app
and PuTTY do not; see the platform scenarios below).

** vim / nvim handle mouse

Both editors are configured with =set mouse=a= (vim) / =opt.mouse = "a"=
(nvim), so they intercept mouse events inside their window.  Vim
additionally sets =ttymouse=sgr= (nvim auto-detects this) -- without it,
vim defaults to =ttymouse=xterm= which doesn't decode tmux's modern SGR
mouse coordinates, and some drags get "stolen" by Terminal.app's own
text selection because vim never sees the events.  A drag enters visual
mode; on release, an =xnoremap <LeftRelease> "+y= mapping auto-yanks the
selection to the ="+= register.  That triggers the =OSC52Yank=
autocmd in vim (or nvim's native clipboard provider) → OSC 52 → tmux
intercept → outer terminal.  End result: *mouse-drag inside vim/nvim
copies to the system clipboard*.

To bypass vim/nvim and use tmux's cross-pane selection instead, hold
*Shift* while dragging -- this gives the terminal/tmux the mouse events,
and tmux's =MouseDragEnd1Pane= binding pipes the selection through
=clip-in= the same way it does at the shell prompt.

** Summary: which layer gets the mouse?

| Context           | Shift held? | Who handles selection      | Where does text go?              |
|-------------------+-------------+----------------------------+----------------------------------|
| In vim/nvim       | No          | vim/nvim (visual mode)     | "+ register via <LeftRelease>    |
| In vim/nvim       | Yes         | Terminal/tmux              | tmux clip-in (system clipboard)  |
| At shell, tmux ON | No          | tmux (pane-aware)          | System clipboard via clip-in     |
| At shell, tmux ON | Yes         | Terminal (raw text)        | PRIMARY (X11) or Windows CB      |
| At shell, tmux OFF| --          | Terminal (raw text)        | PRIMARY (X11) or Windows CB      |

*Tip*: Use tmux selection (no Shift) for most work -- it's pane-aware and goes
to the system clipboard.  Use Shift+drag when you need to select across panes
or when tmux's selection isn't cooperating.

** Unifying mouse behavior across platforms

The default mouse behavior differs across terminal emulators:

| Terminal          | Select to copy?        | Paste action          |
|-------------------+------------------------+-----------------------|
| X11 terminals     | Yes (PRIMARY)          | Middle-click          |
| PuTTY (default)   | Yes (Windows CB)       | Right-click           |
| Windows Terminal  | No (must Ctrl-Shift-C) | Ctrl-Shift-V          |

These can't be made identical (X11 PRIMARY vs system clipboard are
architecturally different), but they can be made close enough for consistent
muscle memory.

*** Recommended terminal settings

*PuTTY*: Window > Selection > Action of mouse buttons: change from
"Compromise" to *"xterm (Right extends, Middle pastes)"*.  Now middle-click
pastes (matching X11 behavior) and right-click extends the selection.

*Windows Terminal*: Settings > Interaction, or in =settings.json=:

#+begin_src json
{
    "copyOnSelect": true,
    "pasteOnRightClick": true
}
#+end_src

This makes Windows Terminal behave like PuTTY's default: select to copy,
right-click to paste.

*X11 terminals*: No change needed.  Select → PRIMARY + middle-click paste is
the native behavior.

*** tmux mouse mode is the real unification layer

Since tmux mouse mode is on by default and tmux intercepts mouse events, the
per-terminal differences only matter when you Shift+drag to bypass tmux.  For
day-to-day work inside tmux, the behavior is already identical everywhere:

- Click-drag in a pane → tmux selects, copies to system clipboard via =clip-in=
- =y= in copy mode → copies to system clipboard via =clip-in=
- =prefix p= → pastes from tmux buffer
- =prefix P= → pastes from system clipboard

* Platform Detection (clip-in / clip-out)

=clip-in= reads stdin and writes to the system clipboard.  =clip-out= reads
the system clipboard and writes to stdout.  They detect the platform in this
priority order:

| Priority | Condition                         | Copy                              | Paste                                                      |
|----------+-----------------------------------+-----------------------------------+------------------------------------------------------------|
|        1 | =$WAYLAND_DISPLAY= + =wl-copy=    | =wl-copy=                         | =wl-paste --no-newline=                                    |
|        2 | macOS (=pbcopy=/=pbpaste=)        | =pbcopy=                          | =pbpaste=                                                  |
|        3 | =$DISPLAY= + =xclip=              | =xclip -selection clipboard -i=   | =xclip -selection clipboard -o=                            |
|        4 | =$DISPLAY= + =xsel=               | =xsel --clipboard --input=        | =xsel --clipboard --output=                                |
|        5 | WSL (=/proc/version= ~ Microsoft) | =clip.exe=                        | =powershell.exe -noprofile -command Get-Clipboard= (+ =tr -d '\r'=) |
|        6 | =$TMUX= set                       | =tmux load-buffer -w -=           | =tmux save-buffer -=                                       |
|        7 | =/dev/tty= exists                 | OSC 52 via =/dev/tty=             | (not supported)                                            |
|        8 | (none)                            | error to stderr                   | error to stderr                                            |

Priority 6 uses =-w= to also emit OSC 52 through tmux.  Priority 7 is the
bare OSC 52 fallback for SSH without tmux -- copy works but paste does not
(OSC 52 read is blocked by most terminals for security).  The WSL paste path
strips trailing =\r= because PowerShell emits CRLF line endings.

All clipboard operations target the CLIPBOARD selection (not PRIMARY).

* Scenarios

** Linux Native (X11) -- e.g. gnome-terminal, xfce4-terminal

=clip-in=/=clip-out= use xclip (or xsel).  Vim's ="+= register works natively
via X11.  tmux OSC 52 also works -- the terminal receives the escape sequence
and sets the X clipboard.  Everything is seamless.

** Linux Native (Wayland) -- e.g. GNOME on Wayland

=clip-in=/=clip-out= detect =$WAYLAND_DISPLAY= and use =wl-copy=/=wl-paste=.
Vim's ="+= register may or may not work depending on build (some builds use
xwayland).  The =clip-in= fallback covers this regardless.  OSC 52 works in
terminals that support it (gnome-terminal, foot).

** macOS -- iTerm2, Ghostty, kitty, Alacritty, WezTerm (full OSC 52 support)

=clip-in=/=clip-out= use =pbcopy=/=pbpaste= (built-in on macOS).  Vim's
="+= register works directly via =pbcopy=/=pbpaste= when vim is built with
=+clipboard= (the default on macOS); otherwise it falls back to =clip-in=.
Over SSH into Linux/BSD, tmux =set-clipboard on= emits OSC 52 sequences that
these terminals translate to the macOS clipboard.  Seamless once the
terminal's OSC 52 permission is granted (see below).

*** Per-terminal OSC 52 permission (important!)

All of these terminals support OSC 52 protocol-wise, but most *gate clipboard
writes behind a setting* by default for security (so a remote process can't
silently overwrite your local clipboard).  Without granting permission, OSC
52 writes are dropped (or you'll see a warning overlay) -- the symptom looks
identical to a terminal that doesn't support OSC 52 at all.

| Terminal   | Setting                                                                |
|------------+------------------------------------------------------------------------|
| iTerm2     | Settings → General → Selection → "Applications in terminal may access clipboard" (enable) |
| Ghostty    | =clipboard-write= defaults to =allow=; set to =ask= or =deny= in =~/.config/ghostty/config= to lock down |
| kitty      | =clipboard_control write= in =~/.config/kitty/kitty.conf= (default allows write) |
| Alacritty  | OSC 52 enabled by default                                              |
| WezTerm    | OSC 52 enabled by default; configurable via =canonicalize_pasted_newlines= and =enable_kitty_keyboard= |

If your test (=printf '\033]52;c;...\a' > /dev/tty; pbpaste=) doesn't return
the test marker, check the terminal setting *before* concluding OSC 52 is
unsupported.

** macOS Terminal.app (built-in) -- *NO OSC 52 support*

Apple's Terminal.app does *not* implement OSC 52 (verified through macOS 15
Sequoia).  Terminal.app silently accepts the escape sequence but never sets
the system clipboard.  Consequences:

- *Locally* (no SSH): clipboard works fine.  =clip-in= matches the =pbcopy=
  branch (priority 2) before any OSC 52 fallback could fire, so tmux
  copy-mode → system clipboard works regardless of Terminal.app's OSC 52
  gap.
- *Over SSH into a remote Linux/BSD host*: remote-to-local clipboard sync
  *is broken*.  On the remote, =clip-in= falls through to =tmux load-buffer
  -w= (priority 6), which emits OSC 52; Terminal.app drops the sequence and
  the data never reaches the macOS clipboard.  Local tmux buffer paste
  (=prefix p=) still works, but =prefix P= / =pbpaste= on the local side do
  not see the remote selection.

See [[*Terminal.app + SSH workarounds][Terminal.app + SSH workarounds]]
below for options.

** Terminal.app + SSH workarounds

If you must use Terminal.app and need remote-to-local clipboard, options:

1. *Switch terminal (recommended)*.  iTerm2, Ghostty, kitty, Alacritty, and
   WezTerm all support OSC 52 and integrate seamlessly with this dotfiles
   setup -- no config changes needed.  All read the =tmux load-buffer -w=
   OSC 52 sequence natively and set the macOS clipboard.

2. *=osc52pty= wrapper*.  https://github.com/roy2220/osc52pty is a third-party
   SSH wrapper that intercepts OSC 52 escape sequences from the remote PTY
   and translates them to local =pbcopy=.  Not included in this dotfiles
   setup, but trivial to adopt: =ssh= calls become =osc52pty ssh ...=.

3. *Manual pull for one-off captures*.  =ssh host "cat file" | pbcopy=
   captures content without any OSC 52 path.  Works for files but not
   interactive tmux copy-mode.

4. *Local-only fallback*.  Accept that remote-to-local clipboard does not
   work and rely on tmux buffers + =prefix p= for in-session paste.  Use
   X11/Wayland forwarding (=ssh -X=) plus =xclip= on the remote if you have
   an X server locally, though this is clunky on macOS (XQuartz required).

Note: =set-clipboard on= in tmux is safe to leave enabled even under
Terminal.app -- the OSC 52 sequence is silently dropped, doing no harm, and
the same config still works correctly when used from iTerm2/Ghostty.

** Windows via PuTTY (SSH to Linux/FreeBSD/OpenBSD)

*PuTTY does not support OSC 52* (as of version 0.83).  There is no setting to
enable it -- the feature is simply not implemented.  This means clipboard
integration over PuTTY is limited to:

- *PuTTY's own mouse selection*: select text in the PuTTY window to copy to
  the Windows clipboard, right-click to paste.  This works but doesn't
  integrate with tmux copy-mode or vim.
- *tmux buffer*: =clip-in= / =clip-out= fall back to =tmux load-buffer= /
  =tmux save-buffer=.  Data round-trips through the tmux buffer but does not
  reach the Windows clipboard.
- *X11 forwarding*: If you enable X forwarding in PuTTY (Connection > SSH >
  X11 > "Enable X11 forwarding") and have an X server running on Windows
  (e.g. VcXsrv, Xming), then =$DISPLAY= will be set and =clip-in= will use
  xclip, which reaches the X clipboard.  This is clunky but functional.

*Recommended alternative*: Use *Windows Terminal* with SSH instead of PuTTY.
Windows Terminal fully supports OSC 52, ships with Windows 10/11, and makes
all clipboard integration (tmux copy-mode, =clip-in=, vim) work seamlessly.
See the "Windows via Windows Terminal" section below.

** Windows via Windows Terminal (SSH to Linux/FreeBSD/OpenBSD)

Windows Terminal supports OSC 52 natively.  No special configuration needed.

There is no =$DISPLAY= on the remote side, so =clip-in= falls through to the
tmux fallback: =tmux load-buffer -w= (the =-w= flag tells tmux to also emit
OSC 52).  tmux has =set-clipboard on= and =allow-passthrough on=, so the
OSC 52 sequence passes through tmux to Windows Terminal, which sets the
Windows clipboard.

tmux copy-mode =y= also works: =copy-pipe-and-cancel 'clip-in'= invokes
=clip-in=, which uses =tmux load-buffer -w=, reaching the Windows clipboard.

** Windows WSL

=clip-in= uses =clip.exe= (built into Windows, accessible from WSL).
=clip-out= uses =powershell.exe Get-Clipboard= (with =\r= stripping).
No =$DISPLAY= or Wayland is needed.  Vim likely lacks =+clipboard= in a
default WSL install, so the =clip-in=/=clip-out= fallback keybindings activate.

** FreeBSD / OpenBSD with X11

Same as Linux Native (X11).  Ensure =xclip= or =xsel= is installed from
packages.  OSC 52 works if the terminal supports it.

** FreeBSD / OpenBSD Console (no X11)

No =$DISPLAY=, no Wayland.  If inside tmux, falls back to tmux buffer
(priority 6).  OSC 52 from tmux may work if the console terminal supports it
(unlikely on raw console, but works over SSH with a capable client -- not
Terminal.app, not PuTTY; see the platform-specific scenarios above).

* Tool Integration

** tmux (=tmux/tmux.conf=)

Mouse mode is on by default (=set -g mouse on=).  Toggle with =prefix m= /
=prefix M=.

| Binding           | Action                                    |
|-------------------+-------------------------------------------|
| =y= (copy mode)  | Yank selection to system clipboard        |
| =M-y= (copy mode)| Yank to system clipboard (with message)   |
| mouse drag end    | Copy selection to system clipboard        |
| =C-y= (copy mode)| Append to =~/tmux.out=                    |
| =prefix C=        | Copy tmux buffer to system clipboard      |
| =prefix P=        | Paste from system clipboard into tmux     |
| =prefix p=        | Paste from tmux buffer (internal)         |
| =prefix m=        | Turn mouse mode on                        |
| =prefix M=        | Turn mouse mode off                       |

OSC 52 (=set-clipboard on=) means tmux also sets the terminal's clipboard
whenever the tmux buffer is updated via copy-pipe.  This is the mechanism that
makes clipboard work over SSH without =$DISPLAY= (requires a terminal that
supports OSC 52 -- Windows Terminal, gnome-terminal, xfce4-terminal, foot,
kitty, Alacritty, iTerm2, Ghostty, WezTerm; *not* PuTTY, *not* macOS
Terminal.app -- see the Terminal.app scenario above).

=allow-passthrough on= lets nested programs (vim, emacs) send their own OSC 52
sequences through tmux to the outer terminal.

** vim (=vim/vimrc=)

Selective clipboard -- yank and paste do NOT automatically touch the system
clipboard.  Explicit keybindings:

| Binding            | Action                                        |
|--------------------+-----------------------------------------------|
| =<leader>y{motion}= | Yank to system clipboard (composable)       |
| =<leader>Y=         | Yank to end of line to system clipboard     |
| =<leader>p=         | Paste from system clipboard (after cursor)  |
| =<leader>P=         | Paste from system clipboard (before cursor) |
| =<leader>C=         | Legacy alias for =<leader>y=                |
| =<leader><space>=   | Copy current line/visual selection to clipboard |

The dotfiles vimrc has a *TextYankPost autocmd* (=OSC52Yank= group) that
fires whenever something is yanked to the ="+= register.  It does two
things, in this order:

1. On macOS, call =pbcopy= directly via =system()=.  This is the
   load-bearing path on macOS Terminal.app and any other terminal where
   OSC 52 doesn't reach pasteboard.  It also defends against vim's
   =clipmethod= machinery short-circuiting the native pasteboard
   provider (observed: with =clipmethod=osc52,wayland,x11=, ="+y= can
   skip the Cocoa pasteboard path entirely).
2. Always emit an OSC 52 escape sequence to =/dev/tty=.  For SSH sessions
   through tmux to a capable outer terminal (Ghostty, Windows Terminal,
   iTerm2, kitty, etc.), this is the load-bearing path -- tmux
   (=set-clipboard on=) intercepts and forwards.  Harmless where
   unsupported (Terminal.app, PuTTY just drop it).

Works on any vim 7.4+ regardless of =+clipboard= or the bundled =osc52=
plugin's terminal-capability check.

Paste is different: OSC 52 read is blocked by most terminals for security,
so we can't symmetrically read the clipboard back over OSC 52.  Paste-side
keybindings (=<leader>p= / =<leader>P=) use the ="+= register only when
vim can actually read from a clipboard provider -- function
=s:HasUsableClipboardPaste()= checks for macOS, =$DISPLAY=, or
=$WAYLAND_DISPLAY=.  Otherwise paste falls back to =clip-out=, which on a
tmux-only Linux remote reads from =tmux save-buffer= -- so =<leader>p=
gives you whatever was last copied through tmux.

You can also use the ="+= register directly at any time:
- ="+ y {motion}= -- yank to clipboard
- ="+ p= -- paste from clipboard
- =:reg "+= -- inspect clipboard register contents

*** Native OSC 52 in vim (9.1+, late 2025)

Vim 9.1+ includes a built-in optional =osc52= plugin that lets vim set the
terminal clipboard directly via OSC 52, without needing =+clipboard= or
=clip-in=.  To enable:

#+begin_src vim
if has('patch-9.1.0') && exists('+clipmethod')
  packadd osc52
  set clipmethod^=osc52
endif
#+end_src

This makes the ="+= register work natively over SSH even on minimal vim builds
without =+clipboard=.  The plugin auto-detects terminal support via DA1 query.
Requires a terminal that supports OSC 52 (not PuTTY).

*Enabled in vimrc* (vim 9.1+ only -- the guard is
=has('patch-9.1.0') && exists('+clipmethod')=) as a *redundant* secondary
path.  In practice, the plugin's emission isn't always exercised: it has a
terminal-capability check that can refuse on chains where the outer
terminal doesn't advertise OSC 52 support (observed: tmux + PuTTY), AND
its clipmethod machinery requires =+clipboard= on some builds (e.g., the
Nix minimal vim build has =+clipmethod= but =-clipboard=, and ="+y= does
not route through clipmethod).

The *load-bearing* path is the =TextYankPost= autocmd (=OSC52Yank= group)
in =vim/vimrc=, which on macOS calls =pbcopy= directly and always emits
OSC 52 to =/dev/tty=.  Works on any vim 7.4+ regardless of vim build
features or terminal-capability detection.

vimrc also configures =clipmethod= conservatively:

- *macOS*: =set clipmethod== (empty) -- forces vim to use the legacy
  =+clipboard= pasteboard path.  vim 9.2's default =clipmethod= on macOS
  is =wayland,x11=, neither of which works on macOS, and vim still routes
  ="+y= through clipmethod first; clearing the option restores
  pre-9.1 native pasteboard behavior.
- *non-macOS*: =set clipmethod^=osc52= so ="+y= emits OSC 52 (intercepted
  by tmux, forwarded to capable outer terminals).

** Spacemacs (=spacemacs/spacemacs=)

The =xclipboard= layer is enabled, which provides:

| Binding    | Action                    |
|------------+---------------------------|
| =SPC x y=  | Copy region to clipboard  |
| =SPC x p=  | Paste from clipboard      |

The layer auto-detects the platform (xsel, clip.exe, etc.) independently of
=clip-in=/=clip-out=.  In terminal mode inside tmux, OSC 52 passthrough also
works if Emacs emits the sequence.

Standard Emacs/Evil clipboard interaction also works:
- ="+y= in evil normal/visual mode (same as vim)
- =M-w= / =C-y= in Emacs mode use the kill ring, which the xclipboard layer
  synchronizes with the system clipboard

** zsh / oh-my-zsh (=_lib/oh-my-zsh/lib/clipboard.zsh=)

oh-my-zsh provides =clipcopy= and =clippaste= functions with their own
platform detection -- *independent of* =clip-in=/=clip-out=, with a
different priority order:

macOS (=pbcopy=) → Cygwin/MSYS (=/dev/clipboard=) → Windows (=clip.exe= +
=powershell.exe=) → Wayland (=wl-copy=) → X11 with =xsel= → X11 with =xclip=
→ =lemonade= → =doitclient= → =win32yank= → Termux → tmux buffer.

Notable differences from =clip-in=: oh-my-zsh tries =xsel= before =xclip=,
prioritizes macOS over Wayland (=clip-in= does the opposite), and has *no*
OSC 52 fallback.  On most platforms (macOS, single-toolchain Linux) the
end-user behavior matches, but the two implementations are not the same code
path.  These are available at the shell prompt:

#+begin_src sh
echo "hello" | clipcopy      # copy stdin
clipcopy somefile.txt         # copy file contents
clippaste                     # paste to stdout
clippaste | grep pattern      # paste and filter
#+end_src

** fzf (=_shell/shellinteractive=)

=ctrl-y= in fzf's file finder and history search copies the selected item to
the system clipboard via =clip-in=.

** clipcmd (=_bin/clipcmd=)

Runs a command, captures output, and places it on the clipboard with a
timestamped header:

#+begin_src sh
clipcmd ls -la    # runs ls -la, copies output to clipboard
#+end_src

Uses =clip-in=/=clip-out= internally.

** clip-test-matrix (=_bin/clip-test-matrix=)

Cross-platform verification script that exercises four clipboard layers in
the current shell environment and reports PASS/FAIL.  Auto-detects platform
and uses the appropriate paste-back mechanism for verification:

*** Expected results -- tmux-only fallback (SSH remote without X11/Wayland)

When the script can't find a local system clipboard (no pbcopy, no =$DISPLAY=,
no =$WAYLAND_DISPLAY=) but =$TMUX= is set, it falls back to verifying via
=tmux save-buffer=.  This is the typical SSH-into-Linux scenario: PuTTY,
Windows Terminal, iTerm2, Ghostty, Terminal.app -- whatever the outer client
is, the *Linux remote* has no system clipboard to read.

*Critical caveat*: in this mode, "PASS" means "data reached tmux's buffer",
*not* "data reached your outer terminal's system clipboard".  tmux intercepts
OSC 52 internally (because =set-clipboard on=) regardless of whether the outer
terminal honors it.  To confirm end-to-end, paste manually in a GUI app on
the real OS after each test.

| Test                | tmux-only verification (any outer terminal) |
|---------------------+---------------------------------------------|
| OSC 52 direct       | PASS (tmux buffer received it)              |
| clip-in script      | PASS (clip-in falls to =tmux load-buffer=)    |
| vim "+ register     | PASS (TextYankPost autocmd in vimrc emits OSC 52 unconditionally; tmux captures) |
| nvim "+ register    | PASS (nvim's clipboard provider auto-picks tmux fallback)  |
| vim OSC 52 forced   | PASS (FP -- via tmux capture, see caveat)   |
| nvim OSC 52 forced  | PASS (tmux captures OSC 52 emission)        |

Note: previously, vim ="+y= without X11 silently failed -- =+clipboard= on
Linux means X11 support, which has nothing to talk to when =$DISPLAY= is
unset; and the bundled =osc52= plugin's terminal-capability check refuses
to emit on tmux+PuTTY.  The dotfiles =vim/vimrc= now installs a
=TextYankPost= autocmd that emits OSC 52 directly, bypassing both gates.
Works on any vim 7.4+ regardless of build features.  See the
=vim (vim/vimrc)= section above.

Whether the outer terminal *also* received the clipboard data:

| Outer terminal           | Receives OSC 52 from tmux?         |
|--------------------------+------------------------------------|
| PuTTY                    | NO -- drops OSC 52 silently        |
| Terminal.app             | NO -- drops OSC 52 silently        |
| Windows Terminal         | YES                                |
| iTerm2                   | YES (if setting enabled)           |
| Ghostty / kitty / WezTerm | YES                                |

So a "PuTTY + tmux + Linux remote" run will show all PASS in tmux-only
mode, but the Windows clipboard will be empty -- the script can't see that
gap.  Test by attempting paste in a Windows app to confirm.

*** Caveats

**** =vim OSC 52 forced= is a false positive when vim has =+clipboard=

On any vim build with =+clipboard= (macOS default, most Linux distros),
the ="+= register writes through the native clipboard provider (pasteboard
on macOS, X11/Wayland on Linux) *in addition to* whatever method
=clipmethod= selects.  Setting =clipmethod=osc52= does emit OSC 52, but the
native provider is also called and the marker reaches the clipboard either
way.  So PASS doesn't prove OSC 52 worked -- the native provider may have
masked an OSC 52 failure.

This is *not* fixable from the test side without disabling =+clipboard= at
the vim build level.  The script prints a =WARN= after the row to flag it.

Authoritative signals for "does OSC 52 work in this terminal":
- =OSC 52 direct= row -- pure terminal layer, no editor in the path.
- =nvim OSC 52 forced= row -- when =vim.g.clipboard= is set to the OSC 52
  provider, nvim *replaces* its clipboard provider entirely; the native
  provider is never called.  Genuine OSC 52 emission, accurate result.

If both of those FAIL, OSC 52 is broken for the current terminal regardless
of what =vim OSC 52 forced= reports.

**** tmux-only mode hides the outer terminal

The script can't introspect what's on the far side of an SSH connection.
In tmux-only mode, every test that produces an OSC 52 sequence will appear
to PASS because tmux's interception buffer captures the data before the
outer terminal even gets a chance to drop it.  Always confirm end-to-end by
pasting in a real GUI app after the run.

* Troubleshooting

- *Clipboard not working over SSH*: Verify your terminal supports OSC 52.
  Known non-supporters: *PuTTY* (as of 0.83) and *macOS Terminal.app* (verified
  through macOS 15 Sequoia) -- switch to Windows Terminal, iTerm2, Ghostty,
  kitty, Alacritty, or WezTerm.  Verify tmux has =set-clipboard on=
  (=tmux show -s set-clipboard=).  Quick local OSC 52 self-test -- copy
  exactly (the base64 string is =test-marker-1234= pre-encoded):
  #+begin_example
  printf '\033]52;c;dGVzdC1tYXJrZXItMTIzNA==\a' > /dev/tty; sleep 0.2; pbpaste
  #+end_example
  If =pbpaste= prints =test-marker-1234=, your terminal supports OSC 52 and
  has clipboard writes enabled.  If it prints something else (or your
  previous clipboard), either (a) the terminal doesn't support OSC 52 at all
  (Terminal.app, PuTTY), OR (b) it supports OSC 52 but clipboard writes are
  gated behind a setting (iTerm2 default, Ghostty default).  Look for a
  warning overlay -- if one appears mentioning "applications in terminal may
  access clipboard", you're in case (b); enable the setting and re-test.
  Run from a fresh terminal (no tmux) for a clean signal; inside tmux, the
  result also depends on =set-clipboard on= + =allow-passthrough on=.
- *Clipboard not working on Wayland*: Install =wl-clipboard= package.
  Verify =$WAYLAND_DISPLAY= is set (=echo $WAYLAND_DISPLAY=).
- *Vim =<leader>y= shells out instead of using register*: Check =vim --version | grep clipboard=.
  If =-clipboard=, vim was built without clipboard support; the fallback to
  =clip-in= is working as intended.  Install a full vim (=vim-gtk3= on Debian,
  =vim-X11= on Fedora) for native register support.
- *tmux =y= only copies to tmux buffer, not system*: Reload config
  (=prefix R=).  Verify =clip-in= works standalone (=echo test | clip-in && clip-out=).
